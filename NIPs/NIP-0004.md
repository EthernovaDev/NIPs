## Abstract

This NIP proposes a fundamental architectural evolution of Ethernova beyond its current EVM+Adaptive Gas model toward a **Layered Deterministic Computer** — a system where applications traditionally requiring centralized servers (email, chat, social media, games, content-addressed browsing) can be built using protocol-native primitives, with final correctness always verifiable deterministically by any public node.

The proposal introduces five new architectural layers on top of the existing EVM: Protocol Objects, Deferred Execution, Protocol Channels, a State Lifecycle system, and Multi-Dimensional Resource Metering. All layers preserve backward compatibility with existing Solidity contracts and Ethereum tooling.

## Motivation

Ethereum claims to be a "world computer" but in practice functions only as a settlement layer. It cannot natively support interactive, stateful, bandwidth-intensive applications due to five structural limitations:

1. **Fully synchronous, atomic execution model** — no concept of sessions, mailboxes, or multi-step interactions at protocol level.
2. **Monolithic, ever-growing state** — all state stored forever, exceeding 200GB and growing unbounded.
3. **One-dimensional gas model** — compute, storage, and bandwidth priced with a single metric, causing systematic mispricing.
4. **No asynchronous communication primitives** — no inbox, outbox, message queue, subscription, or channel at protocol level.
5. **No native content-addressing** — no protocol-level way to reference off-chain content by hash with integrity guarantees.

Ethernova already addresses some of these issues (Adaptive Gas, State Expiry, Anti-MEV, Native Tokens). This NIP proposes the next leap: making Ethernova a genuinely flexible decentralized computer without sacrificing determinism, trust-minimization, or decentralization.

### Design Goals

- Enable decentralized email, chat, social media, games, and browser-like systems as **natural** applications on Ethernova.
- Preserve ~11-second block time and Ethash PoW consensus.
- Preserve full determinism for all on-chain state transitions.
- Keep full node requirements bounded and realistic (8–16 GB RAM, <1 TB disk).
- Maintain backward compatibility — existing Solidity contracts deploy and run without modification.
- Avoid "just move it off-chain to a trusted server" as a solution.

## Specification

### 1. System Vision: Three Truth Layers

Ethernova does not attempt to run all application logic inside block execution literally. Instead, it defines three layers of truth:

| Layer | Location | Determinism | Purpose |
|-------|----------|-------------|---------|
| **Settlement Layer** | On-chain, inside blocks | Fully deterministic | Final state transitions, ownership, proof verification, commitment anchoring |
| **Interaction Layer** | Off-chain, protocol-aware | Cryptographically bound; deterministically resolvable on-chain | Real-time peer interactions using signed messages, channel states, protocol-defined commitment schemes |
| **Witness Layer** | Archival, proof-restorable | Verifiable via Merkle proofs against on-chain roots | Historical/inactive state that can be pruned from nodes but restored with cryptographic proof |

The protocol defines the format, validation rules, and commitment schemes for all three layers. Even though interactions may occur outside blocks, their final correctness is always anchored to the chain and re-verifiable.

### 2. Design Principles

#### Absolute Principles (MUST NOT be violated)

1. **Deterministic Finality**: Every state transition committed to the chain MUST be reproducible identically by all nodes. No ambiguity, no uncommitted randomness.
2. **Cryptographic Verifiability**: Every claim about state — current, historical, or off-chain interaction — MUST be verifiable with cryptographic proofs without trusting the claimant.
3. **Bounded Resource Growth**: Growth of resources required by nodes (disk, RAM, bandwidth) MUST be bounded and predictable. No mechanism may allow unbounded state growth without proportional cost.
4. **PoW Consensus Preservation**: Ethash PoW remains the consensus mechanism. Block time ~11 seconds is unchanged.
5. **EVM Backward Compatibility**: Any Solidity contract valid on Ethereum MUST deploy and execute on Ethernova without modification.

#### Strong Principles (Highly prioritized, may be compromised with justification)

6. **Trust-Minimized Real-Time**: Interactions that appear real-time to users SHOULD be minimally trust-dependent — cryptographic channels preferred over centralized servers.
7. **Ergonomic for Ethereum Developers**: Developers familiar with Solidity/Hardhat/Foundry SHOULD be able to start using Ethernova without drastic learning curve.
8. **Adaptive Resource Economics**: Resource prices SHOULD respond dynamically to network conditions, but changes MUST be smooth and predictable.

#### Compromisable Principles

9. **Universality**: Not all application types need to run with optimal performance. Ethernova prioritizes communication, social, and game state applications — not general-purpose heavy computation.
10. **Full On-Chain Data**: Large data payloads (images, video, files) need not be on-chain. What is on-chain is commitment, routing, and verification. Payloads may be hosted on decentralized layers (IPFS, etc.) with on-chain verified references.

### 3. VM Design: EVM+ with Nova Extension Layer (NEL)

#### 3.1. Architecture Decision

Four options were evaluated:

| Option | Description | Verdict |
|--------|-------------|---------|
| EVM + precompiles only | Minimal changes | Too limited for required primitives |
| **EVM+ with Nova Extension Layer** | EVM extended with new opcodes + precompiles + protocol objects | **Selected** — best balance of compatibility and capability |
| Dual-mode VM (EVM + separate new VM) | Two VMs side by side | Validator complexity doubles; tooling fragments |
| Full NVM replacement | Entirely new VM, backward compatible | Too ambitious; destroys practical compatibility |

#### 3.2. Nova Extension Layer (NEL) Components

NEL adds three categories of extensions:

- **Nova Opcodes**: New EVM instructions for frequently-called operations requiring execution context access.
- **Nova Precompiles**: Contracts at reserved addresses exposing complex protocol functions.
- **Nova Protocol Objects**: First-class entities in the state tree with protocol-managed lifecycles, distinct from standard accounts/contracts.

**Boundary principle:**
- Opcode → operation needs internal execution context access (stack, memory, gas, caller chain) AND is called very frequently (overhead of CALL to precompile unacceptable).
- Precompile → complex operation, less frequently called per-transaction, requires protocol state access.
- Protocol Object → entity with its own lifecycle that must be recognized by consensus (not just data in contract storage).

#### 3.3. Nova Opcodes

##### Category 1: Message & Mailbox Operations

| Opcode | Name | Description |
|--------|------|-------------|
| `0xf6` | `MSEND` | Send message to a Mailbox Protocol Object. Input: target_mailbox_id, payload_hash, metadata. Message enters Pending Queue, committed at end of block. |
| `0xf7` | `MRECV` | Read oldest message from caller's mailbox. Returns: sender, payload_hash, timestamp, sequence_number. Removes message after read. |
| `0xf8` | `MPEEK` | Same as MRECV but read-only (does not remove message). |
| `0xf9` | `MCOUNT` | Return count of unread messages in caller's mailbox. |

**Rationale for opcode (not precompile):** Mailbox operations will be called extremely frequently by contracts implementing chat, email, notifications. CALL + ABI encoding overhead to a precompile is too expensive for high-frequency operations.

##### Category 2: Content Reference Operations

| Opcode | Name | Description |
|--------|------|-------------|
| `0xfa` | `CREF` | Create content reference: input content_hash + size + content_type → returns content_ref_id recorded in protocol. |
| `0xfb` | `CVERIFY` | Verify that a content_ref_id is valid and not expired. Returns: validity, expiry_block, size. |

**Rationale for opcode:** Content referencing is a primitive operation called by nearly every application (social posts, email attachments, game assets). Must be very cheap.

##### Category 3: Session & Channel Operations

| Opcode | Name | Description |
|--------|------|-------------|
| `0xfc` | `SOPEN` | Open a session channel between two parties. Input: counterparty, session_type, timeout_blocks. Returns: session_id. |
| `0xfd` | `SCOMMIT` | Commit state update to session channel. Input: session_id, state_hash, signature. |
| `0xfe` | `SCLOSE` | Close session channel and settle final state to chain. Input: session_id, final_state_hash, both_signatures. |

**Rationale for opcode:** Session management requires execution context access (caller identity, block number for timeouts) and will be used for every channel-based interaction.

#### 3.4. Nova Precompiles (Beyond Existing 0x20–0x28)

New precompiles placed at addresses `0x29`–`0x32`:

| Address | Name | Purpose |
|---------|------|---------|
| `0x29` | `novaMailboxManager` | Mailbox lifecycle management: create, configure (max capacity, retention policy, ACL), destroy. Less frequently called than MSEND/MRECV. |
| `0x2A` | `novaContentRegistry` | Content reference registration and lookup with rich metadata: MIME type, encryption flag, availability proofs, pinning status. A "DNS for content" on Ethernova. |
| `0x2B` | `novaSocialGraph` | Social graph management: follow/unfollow, block, trust score queries, mutual connection checks. Protocol-level shared graph prevents per-dApp fragmentation. Graph traversal too expensive in pure Solidity. |
| `0x2C` | `novaIdentityAttestation` | Identity attestation verification: DID verification, key binding proofs, reputation score queries. Separate from 0x22 (novaAccountManager) — this is identity layer, not account management. |
| `0x2D` | `novaSessionArbiter` | Session channel dispute resolution: if one party disappears or cheats, the other can submit proof for automatic settlement based on last valid state. |
| `0x2E` | `novaContentManifest` | Verification and parsing of content manifests — structured documents describing a decentralized "page" or "application": content reference lists, layout hints, execution permissions, dependency declarations. Primitive for browser-like decentralized systems. |
| `0x2F` | `novaStateWitness` | State witness proof verification: when archived state needs restoration, this precompile verifies Merkle proofs that the state previously existed and is valid. |
| `0x30` | `novaAsyncCallback` | Asynchronous callback registration and resolution: a contract can register a callback triggered in the next block when certain conditions are met. Enables async patterns without nondeterminism — callbacks deterministically scheduled by block height. |
| `0x31` | `novaGameState` | Optimized game state management: commit/reveal schemes, block-hash-anchored VRF, turn validation, compact state diff encoding. Optimized for games needing authoritative state without full game history on-chain. |
| `0x32` | `novaComputeBounty` | Verification of off-chain delegated computation results: a contract posts a computation "bounty," anyone can submit results with proof, precompile verifies proof and releases bounty. Primitive for trust-minimized computation delegation. |

#### 3.5. Nova Protocol Objects

Protocol Objects are first-class entities in the Ethernova state tree, distinct from standard Ethereum Accounts/Contracts. They have their own address space (prefix `0xFF`), protocol-managed lifecycles, and specialized semantics.

##### Protocol Object Types

| Type | Description | Key Fields |
|------|-------------|------------|
| **Mailbox** | Message queue bound to an address | capacity_limit, retention_policy, access_control_list, message_queue |
| **Session** | Bilateral interaction channel | counterparties, state_hash, sequence_number, timeout_block, dispute_rules |
| **Content Reference** | Pointer to off-chain content | content_hash, size, content_type, expiry_block, availability_proof |
| **Identity** | Attestations, key bindings, identity metadata | did_bindings, reputation_score, key_rotation_history |
| **Subscription** | Subscription relationship ("A subscribes to B's feed") | subscriber, publisher, filter_criteria, last_delivered_sequence |
| **Game Room** | Multiplayer game room with contract-defined rules | game_contract, player_list, stake, compact_state, result |

##### Common Fields (All Protocol Objects)

| Field | Type | Description |
|-------|------|-------------|
| `id` | bytes32 | Unique identifier |
| `owner` | address | Owner address |
| `type_tag` | uint8 | Object type discriminator |
| `state_data` | bytes | Type-specific state (size-capped per type) |
| `expiry_block` | uint64 | Block at which object will be archived if not touched |
| `last_touched_block` | uint64 | Last block this object was accessed |
| `rent_balance` | uint256 | Available balance for storage rent payments |

### 4. Execution Model

#### 4.1. Three Execution Modes

##### Mode 1: Synchronous Execution (Ethereum-Compatible)

Identical to Ethereum. Transactions execute inside blocks, atomic, sequential. All legacy Solidity contracts run in this mode. No changes.

##### Mode 2: Deferred Execution

Transactions using MSEND, SCOMMIT, or novaAsyncCallback produce **deferred effects** — committed to the state tree but with side-effects (message delivery, callback execution) occurring in the next block.

**Mechanism:**

1. Transaction in block N calls `MSEND(target, payload_hash)`.
2. Message enters the **Pending Message Queue** in block N's state tree.
3. At the start of block N+1 execution, before regular transactions, the node runs the **Deferred Processing Phase**:
   - All messages in Pending Queue are delivered to target mailboxes.
   - All callbacks scheduled at block N are executed.
   - All session timeouts are checked.
4. Deferred Processing Phase is fully deterministic — ordering determined by insertion order in block N.

**Why this is NOT nondeterministic:** Deferred effects are committed to state at block N (as pending entries). Processing at block N+1 is entirely determined by state at the end of block N. No external input, no randomness. Every node produces identical output.

##### Mode 3: Channel Execution (Off-Chain, Protocol-Bound)

For interactions requiring low latency (real-time chat, game turns), Ethernova defines **Protocol Channels** — similar to state channels, but with protocol-standardized format.

**Mechanism:**

1. Two parties open a channel via `SOPEN` (on-chain transaction, ~11 seconds).
2. After opening, both parties exchange **signed state updates** directly peer-to-peer. State update format is defined by the protocol, not ad-hoc.
3. Each state update has a monotonically increasing sequence number. Both parties store the latest state.
4. At any time, either party can `SCOMMIT` or `SCLOSE` to anchor the latest state to chain.
5. In case of dispute (one party submits old state), `novaSessionArbiter` resolves based on the highest sequence number with valid signatures from both parties.

**How chat feels real-time:**
- User A and B open a channel.
- They exchange signed messages directly P2P. Latency = network latency, not block time.
- Every N messages (or every M seconds), one party commits a state hash to chain as a checkpoint.
- If connection drops or dispute arises, the chain becomes arbiter.
- From user perspective: instant messages. From chain perspective: periodic checkpoint + dispute resolution.

**Trust model:** As long as one party is honest and connected to the network, they can prove the correct state to the chain. This is trust-minimized, not absolutely trustless — but far better than a centralized server.

#### 4.2. Execution Domains

To prevent contamination between execution modes, Ethernova introduces **Execution Domains**:

| Domain | Name | Description |
|--------|------|-------------|
| 0 | Classic | Ethereum-compatible synchronous execution. Default. |
| 1 | Deferred | Execution producing deferred effects. Contracts must explicitly opt-in. |
| 2 | Channel | Contracts participating in Protocol Channels. Different state management rules. |

Contracts declare their domain at deployment. Domain 0 contracts CANNOT call Domain 1 or 2 opcodes (will revert). This prevents unexpected cross-mode interactions.

Cross-domain interaction occurs only through **Domain Bridge Protocol**: Domain 0 contracts can send messages to Domain 1 contracts via MSEND, but responses are deferred (not synchronous). This is by design — forcing developers to think about asynchrony explicitly.

#### 4.3. Determinism Boundary

**Golden rule: Everything entering a block MUST be deterministic. Everything happening in a channel MUST be deterministically resolvable if brought on-chain.**

This means:
- Channel state updates are NOT validated by all nodes in real-time.
- BUT the format, signing scheme, sequence numbering, and dispute resolution rules are defined by the protocol.
- If a dispute occurs, resolution is deterministic — any node validating the dispute transaction will reach the same result.
- "Real-time" does not mean "consensus every millisecond" — real-time means "fast user experience, deterministic settlement."

#### 4.4. Capability-Based Call Model

Ethernova introduces **capabilities** as an extension of the EVM call model.

When a contract calls another contract, in addition to `msg.sender` and `msg.value`, Ethernova provides:
- `msg.capabilities`: a bitmask declaring what the caller is permitted to do (read mailbox, send message, open session, access social graph, etc.)
- Capabilities propagate through the call chain but can only be **narrowed**, never expanded.

This enables:
- Contract A calls Contract B with only "read mailbox" capability — B cannot send messages on A's behalf.
- Library contracts called via DELEGATECALL can have their capabilities restricted.
- Fine-grained permission model without every contract implementing its own access control.

### 5. State and Storage Model

#### 5.1. State Lifecycle Tiers

| Tier | Name | Recency Window | Storage Location | Node Requirement |
|------|------|----------------|------------------|------------------|
| 1 | **Active** | Last 100,000 blocks (~12.7 days) | Hot storage (RAM-accessible, main state trie) | All full nodes MUST have complete Active State |
| 2 | **Warm** | 100,001–1,000,000 blocks ago (~127 days) | Disk, separate trie from Active State | Full nodes maintain Warm State |
| 3 | **Cold** | 1,000,001–10,000,000 blocks ago (~3.5 years) | Disk, Merkle commitment only (root hash, not data) | Full nodes store Merkle root, NOT data. Access requires State Witness. |
| 4 | **Archived** | >10,000,000 blocks ago | Not stored by full nodes | Only historical state roots in block headers remain. Restoration requires proof from archive provider. |
| 5 | **Expired** | Explicitly expired by protocol | Deleted | Owner has grace period (1,000,000 blocks) to pay rent and revive. After grace period, permanently deleted. Only Merkle proof of prior existence remains. |

#### 5.2. State Transition Mechanism

Tier transitions occur **lazily** — not in a single block migrating millions of state entries, but upon access:

1. Transaction attempts to access a storage slot.
2. Node checks current tier based on `last_touched_block`.
3. If state is in a colder tier, transaction MUST pay **warming fee** proportional to tier gap.
4. If state is in Cold/Archived tier, transaction MUST include a **State Witness** (Merkle proof) as part of transaction data.
5. After access, state is promoted back to Active tier and `last_touched_block` is updated.

This process is fully deterministic: tier is determined by `last_touched_block` vs. `current_block`, no ambiguity.

#### 5.3. State Trie Architecture

Ethernova modifies the Ethereum state trie:

```
Global State Root
├── Account Trie (same as Ethereum)
│   ├── EOA accounts
│   └── Contract accounts
│       └── Storage Trie (same, but with last_touched metadata)
├── Protocol Object Trie (NEW)
│   ├── Mailbox Objects
│   ├── Session Objects
│   ├── Content Reference Objects
│   ├── Identity Objects
│   ├── Subscription Objects
│   └── Game Room Objects
├── Pending Effects Queue (NEW)
│   ├── Pending Messages
│   ├── Pending Callbacks
│   └── Pending Session Updates
└── Warm State Commitment Root (NEW)
    └── Merkle root of all Warm State data
```

Each entry in Account Trie and Protocol Object Trie gains additional fields:

| Field | Type | Description |
|-------|------|-------------|
| `last_touched_block` | uint64 | Last block this state was accessed |
| `tier` | uint8 | Current lifecycle tier (computed, not stored — cache only) |
| `rent_balance` | uint256 | Available balance for storage rent (Protocol Objects only) |

#### 5.4. Storage Rent Model

Protocol Objects pay deterministic storage rent:

- **Rent rate**: determined per-byte per-block, adjusted every epoch (10,000 blocks) based on total Protocol Object storage size vs. target size.
- **Rent payment**: can be prepaid at object creation, or topped up at any time by anyone.
- **Rent exhaustion**: when rent_balance is depleted, object enters grace period. If not topped up, object expires.
- **Subsidy**: Contracts can subsidize rent from their own balance — allowing dApp developers to pay storage costs for users.

Regular accounts (EOA/Contract) do NOT pay storage rent — preserving backward compatibility with Ethereum. Only Protocol Objects pay rent. This is a deliberate trade-off: Protocol Objects are opt-in new features where rent can be applied without affecting existing contracts.

However, Contract Storage Slots are still subject to **tier demotion** — slots not accessed for extended periods demote to Warm → Cold → Archived. Difference: demoted slots are not deleted, but future access requires witness proof and warming fee. This provides economic incentive to clean up storage without forcing deletion.

#### 5.5. Merkle Restoration Protocol

When state in Cold/Archived tier needs to be accessed:

1. Caller includes a `StateWitness` in the transaction: a Merkle proof demonstrating the state existed at a specific historical state root.
2. Node verifies witness against historical state root stored in block headers.
3. If valid, state is restored to Active tier.
4. Warming fee is deducted from caller (proportional to tier × data size).

**How users obtain witnesses:**
- **Archive nodes** store all historical state and can generate witnesses on-demand.
- **Witness services**: third parties indexing historical state and providing witnesses as a service (similar to Infura, but for state proofs).
- **Self-serve**: users who store their own data can generate witnesses themselves.

**Security:** A witness can only be valid if the state actually existed. Forging a witness requires finding a hash collision on the Merkle tree — cryptographically infeasible.

### 6. Gas and Resource Economics

#### 6.1. Multi-Dimensional Resource Metering

Ethernova replaces Ethereum's one-dimensional gas model with a **Resource Vector** having five components:

| Dimension | Measures | Analogous To |
|-----------|----------|--------------|
| **Compute** | CPU cycles: arithmetic, logic, control flow | Ethereum gas for non-storage opcodes |
| **State Read** | I/O reads from storage, tier-adjusted | Ethereum SLOAD cost |
| **State Write** | I/O writes + persistence, size and tier proportional | Ethereum SSTORE cost |
| **Protocol Ops** | Operations on Protocol Objects (mailbox send/recv, session ops, content refs) | New — no Ethereum equivalent |
| **Proof Verify** | Witness proof verification, signature batches, cryptographic operations | New — relatively constant cost |

Each transaction has separate limits per dimension:

```
tx.resource_limits = {
    compute:      3,000,000,
    state_read:   1,000,000,
    state_write:    500,000,
    protocol_ops:   200,000,
    proof_verify:   100,000
}
```

Each block also has separate limits per dimension:

```
block.resource_limits = {
    compute:      30,000,000,
    state_read:   10,000,000,
    state_write:   5,000,000,
    protocol_ops:  2,000,000,
    proof_verify:  1,000,000
}
```

#### 6.2. Per-Dimension Adaptive Pricing

Ethernova's existing Adaptive Gas identity is extended to **per-dimension adaptive pricing**:

Each dimension has its own base price, adjusted per block:
- If the previous block used >50% capacity of a dimension, that dimension's base price increases.
- If <50%, base price decreases.
- Adjustment rate: ±12.5% per block (same as EIP-1559, but per dimension).

**Key implication — congestion domain separation:**
- If the network is busy with compute-heavy activity (DeFi arbitrage), compute price rises but protocol_ops price stays low — chat/email does not become expensive because DeFi is busy.
- If the network is flooded with messages (chat spam), protocol_ops price rises but compute price stays stable — DeFi is unaffected.

Different activity categories do not interfere economically.

#### 6.3. Per-Activity Metering

| Activity | Primary Dimension | Pricing Logic |
|----------|-------------------|---------------|
| Pure compute (EVM math) | Compute | Per-opcode, like Ethereum |
| Storage write | State Write | Per-byte + tier adjustment |
| Storage read (Active) | State Read | Per-byte, cheap |
| Storage read (Warm) | State Read | Per-byte × 3 (disk I/O penalty) |
| Storage read (Cold, with witness) | State Read + Proof Verify | Per-byte × 10 + witness verification cost |
| Mailbox send | Protocol Ops | Fixed per-message + per-byte payload |
| Mailbox recv | Protocol Ops | Fixed per-message |
| Session open/close | Protocol Ops + State Write | Fixed + state allocation cost |
| Content ref create | Protocol Ops + State Write | Fixed + size-proportional |
| Social graph update | Protocol Ops | Fixed per-edge |
| Witness verification | Proof Verify | Per-proof-element (Merkle sibling count) |
| Archival restore | State Write + Proof Verify | Restore size + witness cost |

#### 6.4. Backward Compatibility

For transactions not using new features (standard Solidity contracts), the Resource Vector is simplified:
- `gasLimit` from the transaction is mapped to a combination of compute + state_read + state_write.
- Mapping performed by a deterministic formula so existing tooling (MetaMask, ethers.js) continues to function.
- Developers wanting fine-grained control can use an extended transaction format (EIP-2718 style) specifying per-dimension limits.

### 7. Application Primitives

#### 7.1. Decentralized Email

**Architecture:** Mailbox Protocol Object as inbox + Content Reference for body/attachments + Identity Object for addressing.

**Workflow:**
1. Alice writes email body, encrypts with Bob's public key (from Bob's Identity Object), uploads ciphertext to decentralized storage (IPFS, etc.).
2. Alice submits transaction calling `MSEND(bob_mailbox_id, content_hash, metadata)`.
3. In the next block (Deferred Processing), message is delivered to Bob's Mailbox.
4. Bob (or Bob's client) monitors his mailbox. When a new message appears, client retrieves content from decentralized storage using content_hash and decrypts.

**Latency:** ~22 seconds (2 blocks) for delivery. Not instant, but acceptable for email.

**Anti-spam mechanisms:**
- Every MSEND requires gas (Protocol Ops dimension).
- Mailbox owner can set whitelist/blacklist in mailbox ACL.
- Mailbox owner can set minimum "postage" — unknown senders must include a minimum NOVA payment to send.
- Social graph can be leveraged: senders not in the recipient's social graph pay higher postage.
- Whitelisted senders get postage returned (claim-back). Spam is made economically expensive while legitimate communication remains cheap.

#### 7.2. Real-Time Chat

**Direct Messages:** Use Protocol Channels (Mode 3: Channel Execution).

1. Alice and Bob open channel: `SOPEN(bob_address, SESSION_TYPE_CHAT, timeout=100000)` — on-chain, ~11 seconds.
2. After opening, they exchange signed messages directly P2P:
   ```
   message = {
       channel_id: ...,
       sequence: 42,
       sender: alice,
       payload_hash: keccak256(ciphertext),
       timestamp: ...,
       signature: alice_sign(...)
   }
   ```
3. Every 100 messages (or every 5 minutes), one party commits a checkpoint to chain: `SCOMMIT(channel_id, state_hash_at_seq_100, both_sigs)`.
4. User experience: messages sent/received in milliseconds (P2P latency). Deterministic settlement happens in background.

**Group Chat:** Requires a different model since channels are bilateral.

1. A "ChatRoom" contract is deployed in Domain 1 (Deferred).
2. Group members send messages via `MSEND` to the ChatRoom contract's mailbox.
3. ChatRoom contract maintains a subscriber list. During Deferred Processing Phase, the contract fans out messages to all subscriber mailboxes.
4. Latency: ~22 seconds per message (2 blocks).

For lower-latency group chat, a **gossip overlay** is used: group members also communicate P2P while sending commitments to the ChatRoom contract. Messages arrive via P2P (fast) and are confirmed on-chain (slow). Client displays P2P messages with a "pending" indicator until on-chain confirmation.

**Trust model:**
- DM via channel: trust-minimized as long as one party is honest and can submit state to chain.
- Group chat via Deferred: fully trustless (all messages on-chain).
- Group chat via gossip overlay: trust-minimized — messages may be lost in P2P, but any message that existed in P2P can be proven if any party stored the signed message.

#### 7.3. Browser-Like Decentralized Systems

**Concept: Verifiable Content Manifests**

A **Content Manifest** is a structured document describing a decentralized "web page":

```
ContentManifest = {
    manifest_id: bytes32,
    version: uint,
    publisher: address,
    content_refs: [
        { type: "html", ref: content_ref_id_1 },
        { type: "css", ref: content_ref_id_2 },
        { type: "js_safe", ref: content_ref_id_3 },
        { type: "image", ref: content_ref_id_4 }
    ],
    permissions: {
        can_read_identity: false,
        can_send_message: true,
        can_access_social_graph: false
    },
    dependencies: [other_manifest_id_1, other_manifest_id_2],
    signature: publisher_signature
}
```

**How "browsing" works:**
1. Client receives a manifest_id (from a link, social post, search).
2. Client validates manifest on-chain: `novaContentManifest.verify(manifest_id)` — ensuring publisher is valid, content refs are not expired.
3. Client downloads content from decentralized storage using content_refs.
4. Client verifies integrity of each piece of content (hash match).
5. Client renders content according to type.
6. If content is `js_safe`, client executes it in a sandbox with capabilities restricted by manifest.permissions.

**Security:**
- Content manifest has explicit permission model.
- `js_safe` is NOT arbitrary JavaScript — it must be a deterministic subset running sandboxed (similar to WebAssembly execution, not full browser JS).
- Manifests are immutable after publication (content-addressed). Update = new manifest with version increment.
- Client does NOT need to trust the publisher — client independently verifies integrity of all content.

**Limitations (honest trade-off):**
- This is NOT a full "web browser." It is a verifiable content delivery system.
- Interactivity is limited to what the `js_safe` sandbox can do.
- No server-side rendering, no dynamic database queries — all state must come from the chain or be pre-computed.
- This is closer to "static site + chain interaction" than "full web app."

#### 7.4. Social Media

**Available primitives:**
1. **Identity Object**: user profile, display name, avatar content ref, bio.
2. **Social Graph** (via precompile 0x2B): follow/unfollow, block, mutual check.
3. **Mailbox**: for notifications and direct messages.
4. **Content Reference**: for posts (text, images, video — payload off-chain, hash on-chain).
5. **Subscription Object**: so followers receive updates without polling.

**Posting workflow:**
1. User creates content, uploads payload to decentralized storage.
2. User submits transaction: `ContentRef.create(hash, size, type)` + `SocialContract.post(content_ref_id, metadata)`.
3. SocialContract (a Domain 1 contract) receives the post, records it in user's timeline, triggers fanout to subscribers via Subscription Objects.
4. Subscribers receive notification in their mailbox (Deferred Processing).

**Feed construction:**
1. Client queries Social Graph for the user's following list.
2. Client retrieves recent posts from each followed account (from contract state or indexer).
3. Feed rendering happens client-side (not on-chain).

**Moderation:**
- Protocol level: any user can block/mute another address in Social Graph.
- Application level: SocialContract can implement community moderation rules (flagging, reputation-based visibility).
- Content removal: content ref can be expired (owner stops paying rent). Content effectively disappears from chain perspective, though off-chain storage may still retain copies.

**Scalability:**
- Post metadata on-chain, payload off-chain. Storage impact per post: ~200 bytes on-chain (content ref + metadata).
- Social graph at protocol level (not contract storage) — enables optimizations impossible on Ethereum.
- Feed construction delegated to client/indexer — chain provides authoritative data, not rendering.

#### 7.5. Games

**Approach: Authoritative State Anchoring**

1. **Game Room Object** created on-chain: defines game rules (reference to contract), player list, stake (if any).
2. Players interact via **Protocol Channel** (channel execution mode) — turn-by-turn or real-time.
3. Channel state contains game state signed by all players.
4. Periodically (or at game end), final state is committed to chain.
5. Contract validates final state against game rules and distributes rewards/penalties.

**Verifiable Randomness:**
- Games requiring randomness use **commit-reveal** scheme:
  1. Each player submits `hash(random_seed)` at block N.
  2. At block N+1, each player reveals `random_seed`.
  3. Final random = XOR or hash of all revealed seeds.
  4. Players who fail to reveal are penalized (stake slashed).
- For randomness not requiring player participation, Ethernova provides **block-hash-based VRF** via precompile `0x31` (novaGameState).

**State Compression:**
- Only state diffs are committed, not full game state.
- Game Room Object has a compact state format defined by the game contract.
- After game completion, Game Room Object can be archived (demoted to Cold tier) — state effectively "disappears" from Active storage.

**Example — Chess:**
1. Game Room: chess_contract, [alice, bob], stake: 10 NOVA each.
2. Alice and Bob open channel.
3. They exchange moves signed P2P (instant).
4. Every 10 moves, state is committed to chain (checkpoint).
5. Game ends: final state (board position + result) submitted to chain.
6. Chess contract verifies the move sequence is valid.
7. Winner receives 20 NOVA (minus gas).
8. Game Room Object archived.

### 8. Decentralization and Node Requirements

#### 8.1. Node Roles

| Role | Validates Blocks | State Stored | Special Function | Required For Trustlessness |
|------|-----------------|--------------|------------------|---------------------------|
| **Full Node** | Yes, all blocks and transactions | Active + Warm State; Cold/Archived as Merkle commitments only | Runs Deferred Processing Phase; optional P2P channel relay | Yes — anyone can run one |
| **Archive Node** | Yes | ALL historical state (all tiers) | Provides State Witness queries | No — only needed by operators providing witness services |
| **Light Node** | Headers + Merkle proofs only | Block headers and state roots | Verifies transactions via proofs | No — depends on Full Nodes |
| **Channel Relay Node** | No | None | Relays signed P2P channel messages between peers who cannot directly connect | No — cannot falsify messages (all signed); can only censor (user switches relay) |

**Full Node target specifications:**

| Resource | Target |
|----------|--------|
| RAM | 8–16 GB |
| Disk | 500 GB – 1 TB (bounded by state lifecycle) |
| Bandwidth | 10–50 Mbps |
| CPU | 4+ modern cores |

#### 8.2. Bounded Resource Growth

With the State Lifecycle Model:
- Active State size bounded: only state accessed in last ~13 days.
- Warm State bounded: only state accessed in last ~127 days.
- Cold/Archived: only Merkle commitments, constant size per entry (~64 bytes).

**Rough estimate** (Ethereum-equivalent activity level):
- Active State: ~50 GB (vs. Ethereum 200+ GB since everything is active)
- Warm State: ~100 GB
- Cold commitments: ~20 GB
- Total full node: ~170 GB + chain data

Significantly more manageable than Ethereum.

#### 8.3. Efficient Sync

New node synchronization uses **tiered sync**:
1. Sync block headers (fast, like Ethereum fast sync).
2. Download Active State snapshot (largest, but bounded).
3. Download Warm State snapshot.
4. Cold/Archived state: NOT downloaded during sync. Only Merkle roots.
5. Verify: Active + Warm State roots must match state roots in block headers.

Sync time: significantly faster than Ethereum full sync due to smaller data footprint.

#### 8.4. Pruning

Nodes can aggressively prune:
- Warm State demoted to Cold: delete data, keep commitment.
- Expired Protocol Objects: delete after grace period.
- Settled channel state: delete after finality period.

Pruning is fully safe: if data is needed in the future, it can be restored from Archive Nodes via Witness Proof.

### 9. Security and Trust Model

#### 9.1. Attack Surface Analysis and Mitigations

##### Denial of Service (DoS)

| Vector | Mitigation |
|--------|------------|
| Attacker sends many transactions filling one resource dimension | Multi-dimensional metering: attacker must pay in the correct dimension. Spam on protocol_ops is expensive in protocol_ops but does not affect compute. Each dimension has a price floor that rises under spam. |
| Mailbox spam | Mailbox capacity limit + postage requirement + sender whitelist/blacklist. Attacker must pay postage per message — economically infeasible for mass spam. |

##### State Explosion

| Vector | Mitigation |
|--------|------------|
| Attacker creates millions of Protocol Objects with very cheap rent | Rent rate adjusts based on total Protocol Object storage — more objects created = rent rate rises (economic feedback loop). Minimum rent prepayment at creation (must cover at least 10,000 blocks). Per-address creation rate limit for Protocol Objects. |

##### Queue Abuse

| Vector | Mitigation |
|--------|------------|
| Attacker makes Pending Effects Queue very large, slowing block processing | Block-level limit on protocol_ops dimension also limits total pending effects per block. Maximum Pending Queue size in state — if queue is full, MSEND reverts (backpressure). Queue processing time is bounded. |

##### Storage Griefing

| Vector | Mitigation |
|--------|------------|
| Attacker writes data to contract storage and abandons it forever | State Lifecycle Model. Abandoned data demotes to Cold → Archived → only commitment remains. Griefing is self-correcting: unused data eventually stops burdening nodes. |

##### Proof Forgery

| Vector | Mitigation |
|--------|------------|
| Attacker submits fake State Witness to inject state that never existed | Witness verified against historical state roots in block headers. Forging requires finding collision on keccak256 Merkle tree — computationally infeasible. Verification handled by `novaStateWitness` precompile (0x2F). |

##### Channel Replay Attack

| Vector | Mitigation |
|--------|------------|
| Attacker replays old channel state that is more favorable to them | Channel state has monotonically increasing sequence number. `novaSessionArbiter` only accepts state with highest sequence number. Old state is definitively invalid. |

##### Async Race Conditions

| Vector | Mitigation |
|--------|------------|
| Two deferred effects collide — e.g., two messages sent in the same block to a contract reading its mailbox, with results depending on order | Deferred Processing order determined deterministically by transaction index in block. All nodes process deferred effects in exactly the same order. No race condition — only predetermined order. |

##### Capability Escalation

| Vector | Mitigation |
|--------|------------|
| Contract attempts to acquire capability not granted by caller | Capabilities can only be narrowed through call chain, never expanded. Capability bitmask enforced at opcode level (each Nova opcode checks capability before execution). Escalation attempt results in revert. |

##### Cross-Object Inconsistency

| Vector | Mitigation |
|--------|------------|
| Related Protocol Objects (e.g., Mailbox + Subscription) enter inconsistent state | Related Protocol Object mutations batched in a single atomic operation during Deferred Processing Phase. Invariant checks run after each batch. If invariant violated, entire batch reverts. |

##### Upgrade Risk

| Vector | Mitigation |
|--------|------------|
| Protocol changes break backward compatibility or introduce consensus bugs | All new features gated by fork block (like Noven Fork). Domain separation: Domain 0 contracts unaffected by Domain 1/2 changes. Gradual rollout: new features can be disabled per-domain if bugs are found. `novaContractUpgrade` (precompile 0x27) has built-in timelock. |

**Why this is better than "Ethereum with extra opcodes":**

"Ethereum + extra opcodes" is an incremental approach that does not solve structural problems. You could add 100 opcodes to Ethereum and still be unable to build real-time chat, because the fundamental problem is not missing opcodes — the problems are:

1. **No execution model beyond synchronous-atomic** → Deferred Execution Layer solves this.
2. **No asynchronous communication primitives** → Protocol Objects (Mailbox, Session) solve this.
3. **No efficient way for real-time interaction** → Channel Layer solves this.
4. **State grows forever** → State Lifecycle + Witness Layer solves this.
5. **One-dimensional gas causes congestion spillover** → Multi-dimensional resource metering solves this.

Ethernova Next-Gen is not a patched Ethereum. It is a rethinking of what "decentralized computer" means — with the constraint that it must remain compatible with the existing Ethereum ecosystem.

### 12. Honest Trade-Offs

This section is written with full brutality. No sugar-coating.

#### 12.1. What Is Sacrificed

**1. Simplicity.**
The Ethereum EVM is simple — a state machine processing transactions one at a time. Ethernova adds Deferred Processing, Protocol Objects, Channel Layer, State Lifecycle, Multi-dimensional metering. Node codebase complexity increases at minimum 3–4×. Every bug in a new layer is a potential consensus split.

**2. Auditability.**
More protocol-level features = harder to audit. 9 precompiles are already a lot. Adding 10 more + Protocol Objects + Deferred Engine + Channel Arbiter = extremely wide attack surface. A single bug in `novaSessionArbiter` could cause loss of funds from all active channels.

**3. Node operator simplicity.**
Operators must now understand tiers, pruning, witness serving, channel relay. This is more complex than "install geth, run." Barrier to entry for running a node rises, even though hardware requirements remain manageable.

**4. Protocol ossification becomes harder.**
Ethereum can freeze the EVM and focus on L2. Ethernova with this many features will continuously need protocol updates. Ossification (freezing the base layer protocol) becomes far more difficult, conflicting with long-term stability principles.

#### 12.2. What Becomes More Complex

**1. Consensus implementation.**
Every block must run Deferred Processing Phase before Transaction Execution. This adds a critical step to consensus. If two node implementations differ even slightly in deferred processing order, consensus split occurs. This has already happened in Ethernova (GlobalReentrancyGuard bug, binary version mismatch) and the risk multiplies with new features.

**2. Testing.**
The number of possible state transitions explodes. Domain 0 × Domain 1 × Channel settlements × Deferred effects × State tier transitions × Protocol Object lifecycles = combinatorial explosion in test space. Testing becomes an extremely heavy task.

**3. Client development.**
Building a wallet/client for Ethernova becomes far harder than for Ethereum. Clients must handle P2P channel messages, deferred confirmations, Protocol Object queries, witness retrieval. Ecosystem tooling maturity will lag far behind Ethereum for a long time.

#### 12.3. What Is Difficult to Maintain

**1. Determinism across implementations.**
If a second node implementation (e.g., in Rust) is ever created, ensuring identical determinism in Deferred Processing Phase, Protocol Object mutations, and Channel settlement will be extremely difficult. Ethereum already struggles with multi-client determinism at the basic EVM level.

**2. Backward compatibility after breaking changes.**
Every new feature becomes a "forever API." If the Mailbox Object design turns out to be suboptimal, changing it without breaking already-deployed contracts will be very painful. Protocol Objects are more rigid than contracts because their semantics are enforced at the consensus level.

**3. Economic balance.**
Multi-dimensional pricing must be calibrated very carefully. If protocol_ops price is too low, the chain gets spammed with messages. If too high, chat/email becomes non-viable. The right equilibrium may require years of trial and error, and every adjustment risks breaking economic assumptions of existing dApps.

#### 12.4. What Could Make This Design Fail

**1. Nobody uses the new features.**
The most real risk: all new infrastructure is built, but developers only use Domain 0 (basic EVM) because the Ethereum ecosystem is already mature enough and there's no strong incentive to switch. If Protocol Objects and Channel Layer don't gain adoption, they become dead code that adds attack surface without benefit.

**2. Channel Layer trust assumptions don't hold.**
The Channel Layer assumes participants can detect cheating and submit disputes to chain in time. In practice, if a user is offline when the counterparty submits old state, the user can lose funds/data. Watchtower services can mitigate this but add another dependency layer.

**3. State Lifecycle is too aggressive.**
If state that should still be active demotes to Cold tier because thresholds are too low, users will frequently encounter "witness required" — a bad experience. Wrong thresholds can make the chain unusable, while overly conservative thresholds eliminate pruning benefits.

**4. Implementation bugs in the Go fork.**
Ethernova is a CoreGeth fork written in Go. Implementing the Deferred Processing Engine, Protocol Object Trie, Channel Arbiter, and Multi-dimensional Metering in Go — on top of an already complex codebase — is a very large engineering project. Risk of critical bugs is high, and every bug risks breaking consensus.

#### 12.5. Where Flexibility Conflicts with Decentralization

The critical point occurs at **Protocol Object proliferation**:
- Each new Protocol Object type adds complexity to the state trie.
- Too many object types require more RAM for indexing and processing.
- Complex lifecycle management increases CPU time per block.

**Safe boundary:** This design proposes 6 Protocol Object types. Every addition of a new type MUST undergo rigorous evaluation of impact on node requirements. There MUST NOT be any mechanism allowing arbitrary Protocol Object creation — only protocol-defined types.

The second critical point occurs at **Channel Layer bandwidth**:
- If Channel Relay Nodes are incentivized, they require significant bandwidth.
- If the relay network becomes too centralized (only a few large relays), censorship resistance decreases.
- Classic trade-off: flexibility (real-time chat) requires relay infrastructure, which requires incentives, which requires players large enough to be profitable.

#### 12.6. Preventing Overengineering

Practical rules to keep this design grounded:

1. **Ship in phases, not all at once.** Start with Protocol Objects (Mailbox + ContentRef) and Deferred Execution. Channel Layer comes next phase. Multi-dimensional metering after that. Do not attempt to ship everything in one fork.
2. **Measure adoption before adding features.** If Mailbox Objects are unused after 6 months, do not rush to add Social Graph precompile. Validate demand before adding complexity.
3. **Keep Domain 0 sacred.** Whatever happens in Domain 1/2/Channel, Domain 0 (Ethereum compatibility) MUST NOT be affected. This is the escape hatch — if all new features fail, Ethernova still functions as an Ethereum-compatible chain.
4. **Cap Protocol Object types.** Never exceed 8 Protocol Object types without rigorous impact evaluation. Each new type = permanent complexity.
5. **Default to precompile, not opcode.** New opcodes are very expensive in maintenance and cross-client compatibility. Most new functions should start as precompiles. Only if proven to be a bottleneck, promote to opcode.

#### Phase 1 deliverables (as implemented)

Canonical source: [`core/vm/ethernova_protocol_objects.go`](https://github.com/EthernovaDev/ethernova-devnet/blob/72115ac/core/vm/ethernova_protocol_objects.go).

- **Registry address**: `0x000000000000000000000000000000000000FF01` (system storage account; Protocol Object state lives here, distinct from the precompile address `0x29`).
- **Precompile address**: `0x29` (`novaProtocolObjectRegistry`) — dispatched via the stateful-precompile path (`RunStateful`), not the pure `Run` path.
- **Function selectors** (first input byte):
  - `0x01 createObject(typeTag, expiryBlock, rentPrepay, stateData)` — **WRITE** — returns 32-byte ID.
  - `0x02 getObject(id)` — **READ** — returns RLP-encoded `ProtocolObject`, or reverts with `"getObject: not found"`.
  - `0x03 getObjectCount()` — **READ** — returns global live count.
  - `0x04 getObjectsByOwner(owner, offset, limit)` — **READ** — returns length-prefixed array of 32-byte IDs (limit capped at 100).
  - `0x05 deleteObject(id)` — **WRITE**, owner-only.
- **Gas schedule**: create 20,000 · read 2,000 · count 1,000 · list 2,000 · delete 10,000.
- **ID generation**: `keccak256(caller || blockNumber(BE8) || globalNonce(BE8))`. The global nonce is a chain-wide monotonic counter (`keccak256("global_nonce")`) that is NEVER decremented, preventing ID reuse after delete-then-recreate within the same `(caller, block)` context.
- **Storage layout** (all slots keyed under the registry address):
  - `keccak256("object", id)` → `0x01` presence marker.
  - `keccak256("data_len", id)` / `keccak256("chunk_count", id)` / `keccak256("chunk", id, idx)` → chunked RLP payload.
  - `keccak256("owner_count", owner)` → live (non-deleted) count.
  - `keccak256("owner_slots_used", owner)` → monotonic slot high-water mark (iteration covers every slot ever assigned).
  - `keccak256("owner_index", owner, slot)` → object ID (cleared to 0 on delete, leaving a tombstone).
  - `keccak256("owner_slot_of", id)` → reverse lookup used by `deleteObject` for O(1) slot clearing.
  - `keccak256("total_count")` / `keccak256("type_count", typeTag)` → live global/per-type counts.
- **EIP-214 enforcement**: `STATICCALL` routes set `readOnly=true`. For `CALL` / `CALL­CODE` / `DELEGATECALL`, `EVM.inReadOnlyContext()` type-asserts `*EVMInterpreter` and reads its `readOnly` flag, so a contract executing inside a `STATICCALL` frame cannot escape the write ban by doing a zero-value `CALL` into `0x29`. Write selectors (`0x01`, `0x05`) return `ErrWriteProtection` when `readOnly` is set.

#### Deviations from the spec as-written

The spec above describes address `0x29` as `novaMailboxManager`. The devnet implementation consolidates that role into a general-purpose **`novaProtocolObjectRegistry`** at the same address, exposing a uniform `createObject/getObject/getObjectCount/getObjectsByOwner/deleteObject` interface over every Protocol Object type (`typeTag` discriminator in the payload). This directly matches Section 3.5 (Nova Protocol Objects) and Section 5.3 (Protocol Object Trie) — Mailbox / Session / ContentRef / Identity / Subscription / GameRoom all share the same registry. The type-specific precompiles listed for `0x2A`–`0x32` remain reserved and will be added in later phases as specialized facades over (or alongside) the registry.

`types.ProtocolObject` matches §3.5 "Common Fields (All Protocol Objects)" one-to-one: `id`, `owner`, `type_tag`, `state_data`, `expiry_block`, `last_touched_block`, `rent_balance`.

## Roadmap

The design is delivered in phases, consistent with §12.6 ("Ship in phases, not all at once"). Each phase is independently valuable; later phases extend the earlier ones without reopening their on-chain semantics.

| Phase | Scope | Status |
|-------|-------|--------|
| **1. Foundation** | Protocol Object Trie layout (keys, RLP, chunked state, ownership index, global nonce); stateful precompile `0x29` with create/get/count/list/delete; EIP-214 `readOnly` propagation across all call paths. | **Complete** on devnet (`72115ac`) |
| 2. Mailbox semantics | Mailbox type-specific semantics on top of the registry: capacity limits, retention policy, ACL, postage. Initial deferred-message queue data structures. | Pending |
| 3. Deferred Execution Phase | Block-level Deferred Processing Phase: pending-message delivery, scheduled-callback execution, session-timeout checks at the start of block N+1. Opcodes `MSEND` / `MRECV` / `MPEEK` / `MCOUNT`. | Pending |
| 4. Content references & identity | `novaContentRegistry` (`0x2A`) and `novaIdentityAttestation` (`0x2C`) as typed facades over the registry. `CREF` / `CVERIFY` opcodes. | Pending |
| 5. Protocol Channels | `SOPEN` / `SCOMMIT` / `SCLOSE` opcodes; `novaSessionArbiter` (`0x2D`); bilateral signed state update format; dispute resolution. | Pending |
| 6. State Lifecycle | Active / Warm / Cold / Archived tiers; storage rent for Protocol Objects; `novaStateWitness` (`0x2F`) Merkle-proof restoration; tiered sync. | Pending |
| 7. Multi-dimensional metering | Resource Vector (compute / state_read / state_write / protocol_ops / proof_verify); per-dimension adaptive pricing; domain-separated congestion. | Pending |
| 8. Remaining precompiles & Execution Domains | `novaSocialGraph` (`0x2B`), `novaContentManifest` (`0x2E`), `novaAsyncCallback` (`0x30`), `novaGameState` (`0x31`), `novaComputeBounty` (`0x32`); Execution Domains 0/1/2; capability-based call model. | Pending |

Phase ordering is not contractual — phases 4 and 5 can proceed in parallel once Phase 3 is stable. Phase 1 is a strict prerequisite for all subsequent phases because it establishes the Protocol Object Trie every other layer writes into.

## Rationale

The "Layered Deterministic Computer" approach was chosen over alternatives because:

- **Pure L2 approach** would fragment the ecosystem and move trust assumptions off the base layer.
- **Opcode-only extension** does not solve the structural problems (no async primitives, no state lifecycle, single-dimensional gas).
- **Full VM replacement** destroys the Ethereum ecosystem compatibility that gives Ethernova its practical value.
- **The layered approach** allows each layer to be shipped, tested, and adopted independently, with Domain 0 as an always-safe fallback.

The design prioritizes **practical incrementalism** — each layer is independently valuable and does not depend on all other layers being complete. Mailbox + Deferred Execution alone enables decentralized email and notifications. Channel Layer alone enables real-time DMs and games. Multi-dimensional metering alone improves fairness. Together, they form a coherent whole.

## Backward Compatibility

- All existing Solidity contracts deploy and run without modification in Domain 0.
- All existing `eth_*` RPC methods continue to function unchanged.
- All existing tooling (MetaMask, Hardhat, Foundry, ethers.js) continues to work for Domain 0 operations.
- `gasLimit` in standard transaction format is deterministically mapped to Resource Vector dimensions.
- Nova features are strictly opt-in via Domain declaration and new transaction types.

## Security Considerations

See Section 9 for comprehensive attack surface analysis and mitigations covering: DoS, state explosion, message spam, queue abuse, storage griefing, proof forgery, channel replay, async race conditions, capability escalation, cross-object inconsistency, and upgrade risk.

Key security invariants:
- Deferred Processing Phase ordering is fully deterministic (by transaction index).
- Channel dispute resolution is deterministic (highest valid sequence number wins).
- State Witnesses are verified against on-chain Merkle roots (collision-resistant).
- Capabilities can only be narrowed, never expanded, through call chains.
- Protocol Object mutations on related objects are batched atomically with invariant checks.
