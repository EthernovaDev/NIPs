## Summary

A native, privacy-preserving, end-to-end encrypted chat protocol for Ethernova. Uses wallet addresses as identities, smart contracts as channels, and on-chain commitments with off-chain content delivery to prevent state bloat. Designed to leverage Ethernova's unique features (State Expiry, Tempo Transactions, Adaptive Gas, Shielded Pool) to deliver messaging at a cost and scale impossible on Ethereum.

## Motivation

Ethereum is limited to DeFi applications because its architecture makes real-time apps (chat, social media, email) economically impossible. Every on-chain message costs too much, state grows forever, and there's no native privacy.

Ethernova solves these exact problems at Layer 1. A native chat protocol demonstrates the chain's broader thesis: **not just a DeFi chain, but a flexible decentralized computer** that can run the applications Ethereum cannot.

### Why chat first?

- **Universal utility**: Everyone uses messaging daily
- **Proves the thesis**: If chat works on-chain, anything works
- **Leverages all Ethernova features**: State Expiry (old messages), Tempo TX (batching), Shielded Pool (privacy), Frame AA (identity), Adaptive Gas (cheaper storage)
- **No competitor has solved this**: XMTP uses external infrastructure, Farcaster uses hubs, Status uses Waku relays. None are protocol-native.

### Why NOT fully on-chain?

Storing every message on-chain is economically prohibitive:
- 1,000 messages × 200 bytes × 20k gas/byte SSTORE = 4B gas/user/day
- Would break any chain at scale

The solution: **hybrid architecture**. Commitments on-chain, content off-chain.

## Specification

### Architecture Overview

```
+------------------+         +-------------------+         +------------------+
|  User A (wallet) | <-----> |  Ethernova chain  | <-----> |  User B (wallet) |
+------------------+         +-------------------+         +------------------+
         |                          |                             |
         |                    (commitments)                       |
         |                          |                             |
         v                          v                             v
+------------------+         +-------------------+         +------------------+
|  Encrypted blob  |         |   IPFS / blobs    |         |  Encrypted blob  |
+------------------+         +-------------------+         +------------------+
```

**On-chain (cheap):**
- User registry (address → public encryption key)
- Message commitments (hash + metadata)
- Channel/group membership
- Read receipts (optional)

**Off-chain (free/cheap):**
- Encrypted message content
- Attachments (images, files)
- Delivered via IPFS, libp2p relays, or EIP-4844 blobs

### Components

#### 1. novaChatRegistry (Precompile 0x29)

Global user directory mapping wallet addresses to chat identities.

```solidity
interface INovaChatRegistry {
    struct ChatIdentity {
        bytes32 pubkeyX25519;      // X25519 encryption public key
        bytes32 pubkeyEd25519;     // Ed25519 signing public key
        uint64 registeredBlock;
        uint64 lastActivity;
        string metadataURI;         // IPFS URI with display name, avatar, etc.
    }

    function register(bytes32 x25519, bytes32 ed25519, string calldata metadataURI) external;
    function updateMetadata(string calldata metadataURI) external;
    function getIdentity(address user) external view returns (ChatIdentity memory);
    function isRegistered(address user) external view returns (bool);
}
```

**Gas:**
- `register`: 30,000 (one-time)
- `updateMetadata`: 5,000
- `getIdentity`: 2,000

**Storage layout:**
- Key: `address` (20 bytes)
- Value: `ChatIdentity` struct (fixed size for determinism)

#### 2. novaChatChannel (Precompile 0x2A)

Direct messages and group channels.

```solidity
interface INovaChatChannel {
    // Direct message commitment
    function sendDM(
        address to,
        bytes32 messageCommitment,    // keccak256(ciphertext || nonce)
        string calldata contentURI     // IPFS URI or blob reference
    ) external returns (uint256 messageId);

    // Group channel
    function createChannel(
        address[] calldata members,
        bytes32 channelKey,            // shared key encrypted to each member
        string calldata name
    ) external returns (uint256 channelId);

    function sendToChannel(
        uint256 channelId,
        bytes32 messageCommitment,
        string calldata contentURI
    ) external returns (uint256 messageId);

    function addMember(uint256 channelId, address member) external;
    function removeMember(uint256 channelId, address member) external;
    function leaveChannel(uint256 channelId) external;

    // Queries
    function getMessages(address peer, uint256 fromBlock, uint256 limit)
        external view returns (Message[] memory);
    function getChannelMessages(uint256 channelId, uint256 fromBlock, uint256 limit)
        external view returns (Message[] memory);
}
```

**Gas:**
- `sendDM`: 8,000 (leverages Adaptive Gas discount)
- `sendToChannel`: 10,000
- `createChannel`: 50,000 + 3,000 per member
- `addMember` / `removeMember`: 5,000

#### 3. Message Format (off-chain)

Messages are encrypted client-side before being posted to IPFS/blobs:

```
Encrypted message = ChaCha20-Poly1305(
    key = SharedSecret(senderPrivKey, recipientPubKey),  // X25519 ECDH
    nonce = random(24 bytes),
    plaintext = JSON({
        "v": 1,
        "type": "text" | "image" | "file" | "reaction" | "edit" | "delete",
        "content": "...",
        "timestamp": 1234567890,
        "replyTo": "0xHASH" (optional)
    })
)
```

The `messageCommitment` posted on-chain is `keccak256(encrypted_blob || nonce)`, proving message integrity without revealing content.

#### 4. State Expiry Integration

Chat messages leverage NIP-0002's State Expiry:
- Messages older than 90 days → commitments pruned from active state
- User can re-activate old conversations by submitting a Merkle proof
- Prevents unbounded state growth

**Example:** A user with 10,000 messages over 1 year:
- Active state: ~30 days of messages
- Archived state: remaining messages (recoverable with proof)
- State cost: bounded regardless of total message count

#### 5. Tempo Transaction Integration

Batch messaging via NIP-0002's Tempo Transactions:

```solidity
// Send 16 messages in one atomic transaction
TempoTransaction {
    calls: [
        sendDM(alice, hash1, uri1),
        sendDM(bob, hash2, uri2),
        sendToChannel(channel1, hash3, uri3),
        // ... up to 16 calls
    ]
}
```

**Cost savings:** 16 individual messages = 16 × 21,000 intrinsic gas = 336,000.
Tempo batch = 21,000 intrinsic + 16 × 5,000 overhead = 101,000 (70% savings).

#### 6. Privacy Mode (via Shielded Pool)

Optional integration with NIP-0002's Shielded Pool for anonymous messaging:

```solidity
// Anonymous message: hide sender identity
function sendAnonymous(
    bytes32 recipientCommitment,
    bytes32 messageCommitment,
    string calldata contentURI,
    bytes calldata zkProof
) external;
```

Uses commitment-nullifier scheme: sender proves membership in an anonymity set without revealing identity.

### Identity & Frame AA Integration

Chat uses NIP-0002's `novaAccountManager` (0x22) for:

- **Key rotation**: Change encryption key without losing contacts
- **Guardian recovery**: Recover chat account via social recovery
- **Multi-device**: Link multiple devices to one chat identity

### Client Reference Implementation

**Web client** (open source, MIT):
- React + ethers.js
- libsodium-wrappers for X25519/ChaCha20
- IPFS via public gateway (Infura, Pinata)
- WalletConnect for mobile wallet integration

**Mobile** (React Native):
- Push notifications via relayer network
- Biometric unlock
- Offline message queue

### Relayer Network (optional, incentivized)

Light clients don't want to sync the full chain. Relayers provide:
- Message delivery via push notifications
- IPFS content pinning
- Indexing for fast message lookup

**Incentive model:**
- Users pay relayers small NOVA fees per message delivered
- Relayers stake NOVA as anti-spam bond
- Reputation tracked on-chain

## Rationale

### Hybrid architecture (commitments on-chain, content off-chain)

**Considered alternatives:**

1. **Full on-chain**: Rejected. Costs 20k+ gas per message. Breaks at scale.
2. **Full off-chain** (like Waku/XMTP): Rejected. Requires external infrastructure; not protocol-native.
3. **Calldata-only** (no storage): Rejected. Can't query historical messages efficiently.

**Hybrid wins because:**
- Commitments provide integrity, ordering, and censorship resistance
- Content off-chain keeps costs viable
- State Expiry prevents bloat
- Adaptive Gas rewards efficient contracts (chat contracts should be Complex tier, ~+5% penalty)

### Why precompiles instead of Solidity contracts?

Precompiles are 10-100x faster for cryptographic operations and have stable gas costs. A chat registry accessed by every user needs to be:
- Ultra-fast (millions of ops/day)
- Cheap (< 10k gas per query)
- Stable across upgrades

A Solidity contract cannot meet these requirements.

### Why X25519 + ChaCha20 (not AES or secp256k1)?

- **X25519**: Faster than secp256k1 ECDH, resistant to implementation bugs
- **ChaCha20-Poly1305**: Faster than AES on mobile (no hardware accel dependency)
- **libsodium**: Audited, available in every language

## Backwards Compatibility

This NIP introduces new precompiles at addresses 0x29 and 0x2A. These are currently unused and do not conflict with any existing contracts or functionality.

No changes to existing opcodes, gas costs, or consensus rules are required.

## Security Considerations

### Threats & Mitigations

| Threat | Mitigation |
|--------|-----------|
| Metadata leaks (who-talks-to-whom) | Optional Shielded Pool mode |
| Key compromise | Guardian recovery via novaAccountManager |
| Spam | Rate limiting per sender, staking for relayers |
| Malicious content | Client-side content filters (IPFS content addressable) |
| IPFS unavailability | Multiple gateways, blob storage fallback |
| Replay attacks | Nonces in message commitments |
| Forward secrecy | Ephemeral keys per session (optional double ratchet) |

### Known Limitations

- **IPFS reliability**: Content may become unavailable if not pinned. Mitigation: incentivized pinning via relayer network.
- **Metadata privacy**: On-chain commitments reveal timing and participant addresses (mitigated by Shielded Pool).
- **Mobile UX**: Requires wallet integration (worse UX than Telegram/Signal).

## Test Cases

1. **Registration**: User registers identity, retrieves it, verifies keys match.
2. **DM flow**: Alice sends message to Bob, Bob decrypts, verifies commitment.
3. **Group chat**: Create 5-member channel, all members can send/receive.
4. **Key rotation**: User rotates keys, old messages still readable.
5. **State expiry**: Message from >90 days ago is archived, recoverable with proof.
6. **Tempo batch**: 16 messages sent atomically, all succeed or all revert.
7. **Anonymous mode**: Message sent via Shielded Pool, recipient receives without knowing sender.

## Research Questions (open for discussion)

1. **Message retention**: Should default expiry be 30/60/90/180 days?
2. **Anonymity set size**: Minimum k-anonymity for Shielded Pool mode?
3. **Group size limits**: Max members per channel? (100? 1000? unlimited?)
4. **Moderation**: How to handle abuse reports in a decentralized manner?
5. **Search**: On-chain indexing vs client-side search with encrypted indexes?
6. **Cross-chain**: Should chat identities be portable to other EVM chains?
7. **Incentive model**: Should senders pay recipients (anti-spam), or is gas cost enough?
8. **EIP-4844 blobs**: Use blob space for message content (cheaper than calldata)?

## Inspiration

- **XMTP**: Wallet-to-wallet messaging (but uses external infrastructure)
- **Signal**: E2E encryption (double ratchet, X3DH)
- **Matrix**: Federated chat protocol
- **Nostr**: Event-based decentralized messaging
- **Farcaster**: Social protocol with hubs
- **Waku**: Messaging layer for Ethereum

Ethernova Chat combines the best of these: native to the protocol (unlike XMTP), fully decentralized (unlike Farcaster hubs), efficient (unlike fully-on-chain approaches), and privacy-preserving (via Shielded Pool).
